# 오픈소스 스택과 아키텍처 재검토 (v2)

> **재검토 근거**: 제텔카스텐 + Karpathy LLM Wiki + A-MEM + LightRAG + Notion 벡터 RAG 분석을 바탕으로 기존 v1 아키텍처의 구조적 결함을 수정한 변경안
> **원본 문서**: [1. 오픈소스 스택과 아키텍처 검토](https://www.notion.so/33d2f5d38f228103bee1d0194d8c5f3d)

## 1. v1의 구조적 결함 진단

기존 아키텍처는 **수집은 잘 되지만 지식으로 전환되지 않는** 구조다.

| 문제 | v1 상태 | 원인 |
|------|---------|------|
| 수집→저장 사이 지식 연결 레이어 없음 | 수집 자료가 원시 데이터로만 쌓임 | Karpathy가 지적한 "죽은 데이터" 문제 |
| 노트 간 연결이 수동 | Obsidian 백링크를 사람이 직접 관리 | 규모가 커질수록 불가능 |
| 벡터 DB 도입을 "중기"로 미룸 | 초기에는 연결 인프라 전무 | 시작부터 아일랜드 문제 발생 |
| Chroma 단독 채택 | 그래프 관계 없이 벡터 유사도만 | 창발적 연결(다중 홉 추론) 불가 |
| 분석 레이어 = GPT Researcher 1개 | 리포트 생성에만 집중 | 노트 간 자동 연결 기능 없음 |
| Notion 공유 = 단순 게시 | n8n → Notion API 푸시 | Notion 내장 벡터 RAG 미활용 |

> **핵심**: 수집 파이프라인은 훌륭하지만 수집된 자료가 **서로 연결되지 않아 창발이 일어나지 않는다**.

## 2. 재검토 근거 — 핵심 인사이트 3가지

### 인사이트 1: Karpathy 방식의 한계와 frontmatter 해법

Karpathy: raw/ 폴더 → LLM이 40만 단어 전체 읽기 → 위키 생성
문제: 문서 1,000개 이상에서 토큰 폭탄. 일반 사용자에게 적용 불가.

바이브랩스 해법:
- `Archive 플래그` → 처리된 자료를 숨겨 inbox를 항상 비워둠
- `frontmatter 캐싱` → 각 노트에 summary+tags+id 저장, 본문 읽기 없이 연관성 판단
- 토큰 사용량 **10분의 1**로 절감

### 인사이트 2: LightRAG가 Chroma를 대체한다

| | Chroma (v1) | LightRAG (v2) |
|-|-----------|-------------|
| 검색 방식 | 벡터 유사도만 | 그래프 순회 + 벡터 하이브리드 |
| 다중 홉 추론 | 불가 | 자연스럽게 지원 |
| 새 자료 추가 | 재인덱싱 필요 | 증분 업데이트 |
| 창발적 연결 | 제한적 | 엔티티 관계망으로 강력 |

LightRAG는 내부적으로 벡터 저장 포함 → Chroma 별도 불필요. (EMNLP 2025, GitHub 30,000+ 스타)

### 인사이트 3: Notion 자체가 벡터 RAG로 작동한다

Notion 공식 블로그: *"Two years of vector search at Notion: 10x scale, 1/10th cost"*

공유 레이어에서 Notion을 사용하면 **검색 인프라를 별도 구축할 필요 없음**. 팀이 Notion AI 검색으로 모든 수집 자료를 시맨틱 검색 가능.

│                      TRIGGER LAYER                          │
│       n8n 크론 스케줄 (자동)  │  Claude MCP 온디맨드 (수동)  │
└──────────────────┬────────────────────────┬─────────────────┘
                   │                        │
┌──────────────────▼────────────────────────▼─────────────────┐

│  기독교: RSS 14개, scholarly, Firecrawl(pts.ac.kr)          │

                               │
┌──────────────────────────────▼──────────────────────────────┐

│  • frontmatter: id, tags, summary, source, date             │
│  • 처리 완료 → raw/archive/ 이동 (Inbox Zero)               │
└──────────────────────────────┬──────────────────────────────┘
                               │
┌──────────────────────────────▼──────────────────────────────┐
│               ✅ KNOWLEDGE LINK LAYER [신규]                 │
│  Zettelkasten MCP → Claude가 직접 노트 읽기/연결            │
│  • supports / contradicts / extends / inspired-by 링크 타입 │

                               │
┌──────────────────────────────▼──────────────────────────────┐
│                   ANALYSIS LAYER                            │
│  GPT Researcher → 주간 클러스터 리포트 자동 생성             │
│  LightRAG → 그래프 순회 기반 다중 홉 Q&A                    │
│  Claude MCP → 온디맨드 요약·번역·도메인 간 연결 발굴        │
└──────────────────────────────┬──────────────────────────────┘
                               │
┌──────────────────────────────▼──────────────────────────────┐
│                    STORAGE LAYER                            │
│  Obsidian Vault (Zettelkasten 구조)                         │
│  ├── inbox/       ← 원자화 완료, 연결 대기                  │
│  ├── literature/  ← 출처 있는 문헌 노트                     │
│  ├── permanent/   ← 연결된 연구 노트 (핵심 지식)            │

                               │
┌──────────────────────────────▼──────────────────────────────┐
│                    SHARING LAYER                            │
│  n8n → Notion API (publish:true 노트만 선별 동기화)         │
│  Notion 내장 벡터 RAG → 팀 시맨틱 검색 활용                │

## 5. 변경된 오픈소스 스택 상세

### 5-1. 원자화 레이어 — 신규 추가

| 도구 | 역할 | 비고 |
|------|------|------|
| **Claude Code 스킬** `/literature` | 수집 자료 → frontmatter 포함 원자적 노트 | `.claude/sys/literature.md` |
| **Claude Code 스킬** `/fleeting` | 단편 메모·아이디어 즉시 기록 | `.claude/sys/fleeting.md` |
| **n8n HTTP 노드** | 수집 완료 이벤트 → 스킬 자동 호출 | 기존 n8n 워크플로우 확장 |

**frontmatter 표준 스키마**:

```yaml
---
id: "LIT-2026-0410-001"
title: "제목"
source: "https://..."
date: "2026-04-10"
domain: [학술|뉴스|커뮤니티|소셜|경제|기독교]
tags: [AI, 제텔카스텐]
summary: "30자 이내 한줄 요약"
links: []
status: inbox
---
```

### 5-2. 지식 연결 레이어 — 신규 추가

| 도구 | 역할 | 출처 |
|------|------|------|
| **Zettelkasten MCP** | Claude 직접 통합, 의미 링크 타입 정의 | entanglr/zettelkasten-mcp |
| **Claude Code 스킬** `/permanent` | 두 노트 연결 → 새 연구 노트 생성 | 인간이 자기 말로 정제 |
| **Claude Code 스킬** `/lint` | 고아 노트 탐지, 끊긴 링크 수정 | 주 1회 자동 실행 |
| **Claude Code 스킬** `/index` | 클러스터별 인덱스 + 관계도 생성 | 노트 50개 추가 시 갱신 |

**링크 주석 방식 (A-MEM 적용)**:

```
[[노트A]] —(supports)→ [[노트B]]
  주석: "A의 frontmatter 캐싱 전략이 B의 토큰 절약 주장을 실증함"
→ 이 주석이 나중에 새 permanent 노트의 씨앗이 됨
```

### 5-3. 검색 인프라 — Chroma → LightRAG 교체

| | v1: Chroma | v2: LightRAG |
|-|-----------|------------|
| **도입 시점** | 중기 (수만 건) | **초기부터** |
| **검색 방식** | 벡터 유사도 | 그래프 + 벡터 하이브리드 |
| **다중 홉 추론** | 불가 | 지원 |
| **증분 업데이트** | 재인덱싱 필요 | 지원 |
| **내부 벡터 저장** | 직접 관리 | 내장 (별도 Chroma 불필요) |

```bash
# LightRAG Docker 설치
docker run -d --name lightrag \
  -p 9621:9621 \
  -v ./lightrag_data:/app/data \
  lightrag/lightrag:latest
```

적용 범위: Obsidian `permanent/` 폴더 전체를 LightRAG에 인덱싱

### 5-4. 분석 레이어 — GPT Researcher 역할 확장

| 도구 | v1 역할 | v2 역할 |
|------|---------|---------|
| **GPT Researcher** | 주간 트렌드 리포트 | 유지 + **클러스터 연결 리포트** 추가 |
| **LightRAG** | 없음 | **다중 홉 Q&A** 백엔드 신규 |
| **Claude MCP** | 요약·번역·분류 | 유지 + **도메인 간 연결 발굴** 추가 |

### 5-5. 공유 레이어 — Notion RAG 적극 활용

```
v2 워크플로우:
  n8n 주간 트리거
  → permanent/ 폴더 스캔 (publish: true 필터)
  → Notion API로 동기화
  → 팀원이 Notion AI로 "물리학과 신학의 교차점" 검색
  → Notion 내장 벡터 RAG가 관련 노트 자동 추출
```

## 6. 확장성 전략 재설계

### v1 문제 (단계 간격이 너무 큼)

```
초기(수천 건) → Dataview + SQLite  ← 연결 없음
중기(수만 건) → Chroma 도입         ← 너무 늦음
장기(수십만 건) → GraphRAG
```

### v2 수정안

| Phase | 규모 | 핵심 인프라 | 연결 방식 |
|-------|------|-----------|---------|
| **1** | ~200 노트 | Claude Code 스킬 + frontmatter | /permanent 스킬 반수동 |
| **2** | ~1,000 노트 | LightRAG Docker | 그래프 기반 자동 연결 후보 |
| **3** | ~5,000 노트 | LightRAG + Zettelkasten MCP | 완전 자동 연결 + 주석 생성 |
| **4** | 10,000+ 노트 | GraphRAG (Microsoft) | 온톨로지 자동 구성 |

## 8. 변경 전후 한눈에 비교

```
[v1] 수집 → 분석(GPT Researcher) → 저장(Obsidian) → 공유(Notion)
          문제: 수집과 저장 사이에 지식 연결 없음. 창발 불가.

[v2] 수집 → 원자화(Claude 스킬) → 연결(Zettelkasten MCP)
          → 검색(LightRAG) → 분석(GPT Researcher + 연결 리포트)
          → 저장(Obsidian Zettelkasten) → 공유(Notion 벡터 RAG)
          개선: 수집된 즉시 원자화 → 자동 연결 → 창발 가능한 구조
```
