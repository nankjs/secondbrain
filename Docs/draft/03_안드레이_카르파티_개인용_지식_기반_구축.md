# 안드레이 카르파티: 개인용 지식 기반 구축

---

## Karpathy 원문 트윗 (2026년 4월 3일)

> **안드레이 카르파티** @karpathy · 2026년 4월 3일 · [링크 🔗](https://x.com/karpathy/status/2039805659525644595)
>
> **LLM 지식 기반**
>
> 최근 제가 매우 유용하다고 느끼는 점: LLM을 활용해 연구 관심 분야의 다양한 주제에 대한 개인 지식 기반을 구축하는 것입니다. 이렇게 함으로써, 최근 제 토큰 처리량의 상당 부분이 코드 조작보다는 지식(마크다운 및 이미지로 저장됨) 조작에 더 많이 쓰이고 있습니다. 최신 LLM들은 이 작업을 꽤 잘 수행합니다. 따라서:
>
> **데이터 수집**: 소스 문서(기사, 논문, 저장소, 데이터셋, 이미지 등)를 raw/ 디렉터리에 색인화한 다음, LLM을 사용하여 위키를 점진적으로 "컴파일"합니다. 이 위키는 단순히 디렉터리 구조 내의 .md 파일 모음입니다. 이 위키에는 raw/ 디렉토리의 모든 데이터 요약과 백링크가 포함되며, 데이터를 개념별로 분류하고 각 개념에 대한 글을 작성한 뒤 모두 연결합니다. 웹 기사를 .md 파일로 변환할 때는 Obsidian Web Clipper 확장 프로그램을 사용하는 것을 선호하며, 핫키를 사용하여 관련 이미지를 모두 로컬에 다운로드해 LLM이 쉽게 참조할 수 있도록 합니다.
>
> **IDE**: 저는 원시 데이터, 컴파일된 위키, 그리고 파생된 시각화 자료를 볼 수 있는 IDE "프론트엔드"로 Obsidian을 사용합니다. 중요한 점은 LLM이 위키의 모든 데이터를 작성하고 관리하며, 제가 직접 손을 대는 경우는 거의 없다는 것입니다.
>
> **Q&A**: 흥미로운 점은 위키가 충분히 커지면(예: 최근 연구 관련 제 위키는 약 100개의 문서와 40만 단어 규모입니다), LLM 에이전트에게 위키를 기반으로 온갖 복잡한 질문을 던질 수 있고, 에이전트가 스스로 답을 찾아 연구하는 등의 작업을 수행한다는 것입니다. 처음에는 복잡한 RAG(Retrieval-Augmented Generation) 방식을 써야 할 줄 알았지만, LLM이 인덱스 파일과 모든 문서의 간략한 요약을 자동으로 잘 관리해 주었고, 이 정도 '소규모'에서는 중요한 관련 데이터를 꽤 쉽게 읽어내더군요.
>
> **출력**: 텍스트나 터미널로 답변을 받는 대신, 마크다운 파일이나 슬라이드 쇼(Marp 형식), matplotlib 이미지를 생성해 주는 것을 선호하며, 이 모든 결과물을 Obsidian에서 다시 확인합니다.
>
> **린팅(Linting)**: 위키에 대해 몇 가지 LLM "상태 점검"을 수행하여, 예를 들어 불일치하는 데이터를 찾거나, 누락된 데이터를 (웹 검색기를 통해) 보완하거나, 새로운 기사 후보를 위한 흥미로운 연관성을 찾는 등, 위키를 점진적으로 정리하고 전반적인 데이터 무결성을 강화했습니다.
>
> **요약**: 특정 수의 소스에서 원시 데이터를 수집한 후, LLM이 이를 .md 위키로 컴파일하고, LLM이 다양한 CLI를 통해 이를 조작하여 Q&A를 수행하고 위키를 점진적으로 개선하며, 이 모든 내용을 Obsidian에서 볼 수 있게 합니다. 위키를 수동으로 작성하거나 편집하는 일은 거의 없으며, 이는 전적으로 LLM의 영역입니다. 저는 이 분야에 스크립트를 엮어 만든 임시방편적인 솔루션 대신, 놀라운 신제품을 만들 여지가 있다고 생각합니다.

## 핵심 전환: 코드에서 지식으로

토큰의 용도가 달라졌습니다.
Karpathy의 최근 토큰 처리량 대부분은 코드 조작이 아니라 **지식 조작**에 쓰이고 있습니다.
논문, 아티클, 레포, 데이터셋 — 쌓기만 하면 죽은 데이터입니다.
정리하고, 연결하고, 검색 가능하게 만드는 것이 진짜 작업이거든요.
LLM이 이 전체 과정을 자동으로 수행한다는 게 이 워크플로우의 핵심입니다.

---

## 6단계 워크플로우

### 1/ 데이터 수집 — raw/ 디렉토리

원본 자료(아티클, 논문, 레포, 데이터셋, 이미지 등)를 `raw/` 디렉토리에 색인화합니다.
- **웹 아티클** → [Obsidian Web Clipper](https://obsidian.md/clipper) 확장 프로그램으로 `.md` 파일 변환
- **관련 이미지** → 핫키로 로컬 다운로드하여 LLM이 쉽게 참조 가능
- 수집 단계는 사람이 합니다. 진짜는 다음 단계부터입니다.

### 2/ 컴파일 — LLM이 위키를 만든다

LLM이 `raw/`를 읽고 디렉토리 구조 내의 `.md` 파일 모음으로 된 **위키를 점진적으로 "컴파일"** 합니다.
위키에는 세 가지가 들어갑니다:
- `raw/` 디렉토리의 **모든 데이터 요약과 백링크**
- 데이터를 **개념별로 분류**하고 각 개념에 대한 **아티클 작성**
- 모든 문서를 **상호 연결(interlinking)**

> **핵심**: LLM이 위키의 모든 데이터를 직접 쓰고 유지합니다. 사람은 거의 건드리지 않습니다.

### 3/ IDE — Obsidian이 프론트엔드

Obsidian을 IDE "프론트엔드"로 사용하여 세 가지를 한 곳에서 봅니다:
- **원시 데이터** (`raw/`)
- **컴파일된 위키** (`.md` 파일 모음)
- **파생된 시각화** (슬라이드, 차트 등)

플러그인 활용:
- **Marp** — 슬라이드 프레젠테이션 렌더링
- **matplotlib** — 데이터 시각화 이미지 생성

### 4/ Q&A — 위키가 커질수록 강력해진다

위키가 **~100개 문서, ~40만 단어** 규모가 되면 LLM 에이전트에게 온갖 복잡한 질문을 던질 수 있습니다.

> **RAG가 필요할 줄 알았는데 아니었습니다.**
> LLM이 **인덱스 파일과 모든 문서의 간략한 요약**을 자동으로 관리해 주었고, 이 정도 '소규모'에서는 관련 데이터를 꽤 쉽게 읽어냅니다.

**출력 형식은 텍스트가 아닙니다:**
- `.md` 마크다운 파일
- Marp 형식 슬라이드쇼
- matplotlib 이미지

결과물을 다시 위키에 "파일링"합니다. 탐색과 질문이 쌓일수록 지식 베이스가 **축적·강화**되는 구조입니다.

### 5/ 린팅(Linting) — 위키의 자가 치유

LLM이 위키에 대해 **"상태 점검"**을 수행합니다:
- **불일치하는 데이터** 발견 및 수정
- **누락된 데이터**를 웹 검색을 통해 보완
- 새로운 기사 후보를 위한 **흥미로운 연관성** 발굴
- 추가로 물어보고 조사해 볼 만한 **질문 제안**

### 6/ 추가 도구 & 확장

- 위키용 소규모 **검색 엔진**을 직접 코딩
- 리포지토리가 커짐에 따라 **합성 데이터 생성 및 파인튜닝**으로 데이터를 컨텍스트 윈도우가 아닌 가중치 내에 새기는 방향도 탐구

| 단계 | 입력 | 처리 | 출력 |
|------|------|------|------|
| 수집 | 아티클, 논문, 레포, 이미지 | Obsidian Web Clipper + 핫키 | `raw/` 디렉토리 |

## 왜 RAG가 아닌가?

VentureBeat는 이 접근법을 **"RAG를 우회하는 LLM 지식 베이스 아키텍처"**라고 명명했습니다.

기존 RAG의 한계:
- 문서를 임의의 "청크"로 잘라 벡터로 변환 → 유사도 검색
- 맥락이 깨지고, 복잡한 인프라(벡터 DB)가 필요

Karpathy 방식의 차별점:
- LLM이 **인덱스 파일과 문서 요약을 자동 유지**
- 구조화된 마크다운 텍스트에 대한 **추론 능력**에 의존
- 중간 규모(~40만 단어)에서는 벡터 검색 없이도 관련 데이터를 정확히 찾아냄
- **100% 검사 가능(inspectable)** — 블랙박스가 아닌 투명한 `.md` 파일

## 핵심 요약

**Karpathy 방식의 3가지 원칙:**

1. **사람은 수집만, LLM이 정리한다** — 위키 작성·유지·린팅 모두 LLM의 영역
2. **RAG 없이 추론으로 충분하다** — 중간 규모에서는 인덱스 파일 + 구조화된 마크다운만으로 작동
3. **탐색이 축적된다** — 질문과 결과가 위키에 다시 파일링되어 지식 베이스가 자가 성장

## 대화에서 발굴한 통찰 — Karpathy × 루만 × 토큰 경제학

### 1/ "점진적 위키 빌드"의 실체

Karpathy가 말한 "LLM이 점진적으로 위키를 컴파일한다"는 것은 **incremental build**와 같은 개념이다. `raw/`에 새 파일이 추가되면 전체를 다시 쓰는 게 아니라, **변경된 delta만 처리**하여 기존 위키 문서를 업데이트하고 백링크를 삽입하는 구조다.

### 2/ 새 파일 감지 — LLM 혼자서는 불가능하다

LLM 자체는 파일 시스템을 모니터링하지 못한다. 새 파일 감지는 두 가지 방식:
- **인덱스 파일 diff** — `index.md`에 기록된 처리 완료 목록과 `raw/`의 실제 파일 목록 비교
- **셸 명령 전처리** — `find raw/ -newer wiki/.last_compiled` 또는 `git diff --name-only`로 변경 파일 필터링 후 LLM에 전달

### 3/ 토큰 특권 — 일반 사용자는 Karpathy처럼 쓸 수 없다

Karpathy의 워크플로우는 **토큰 무한 공급**을 전제로 한다. OpenAI 공동창립자이기에 가능한 비용 구조이며, 일반 사용자에게는 **같은 효과를 1/10 토큰으로 내는 설계**가 필요하다.

### 4/ 제텔카스텐의 세 메모 유형과 Karpathy 방식의 대응

| 제텔카스텐 | Karpathy | 세컨드 브레인(Notion) |
|----------|---------|-------------------|
| 임시메모(Fleeting) | `raw/` 디렉토리 | In Notes DB |
| 문헌메모(Literature) | raw/ 내 요약+해석 | In Notes DB 분석 영역 |
| 영구메모(Permanent) | 컴파일된 위키 .md | Thinker Notes DB |

핵심 차이: 루만은 임시메모를 처리 후 **폐기**했고, Karpathy는 `raw/`를 **영구 보존**한다. 디지털 시대의 해법은 **아카이브 플래그** — 삭제하지 않되, 활성 탐색 공간에서는 제외한다.

### 5/ 통합 설계 — 세 시스템을 넘어서는 아키텍처

**핵심 공식:** 임시메모 → 처리 완료 → Archive 플래그 → raw/는 늘 비어있음 → 영구메모에 아이디어 집약 + 임시메모 레퍼런스 보유 → 프론트매터(DB 프로퍼티)에 요약 캐싱 → LLM 토큰 절약

이 구조가 해결하는 세 가지 문제:
1. **Inbox Zero** — 처리된 임시메모가 Archive로 빠지므로, 미처리 메모 판별 비용 = 0
2. **분산 인덱스** — 각 영구메모의 프론트매터가 요약을 들고 있음. LLM이 프로퍼티만 읽으면 전체 맥락 파악 가능
3. **레퍼런스 체인** — 영구메모 → 임시메모(Archive) 링크가 살아있어 원본의 독자적 생각도 언제든 추적 가능

### 6/ 유튜브 킬러 앵글

> *"Karpathy는 raw/를 쌓고, 루만은 임시메모를 버렸다. 우리는 아카이브하고 링크한다. 그리고 프론트매터로 LLM의 토큰을 1/10로 줄인다."*
