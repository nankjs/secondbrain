# 세컨드브레인 지식관리시스템 고도화 로드맵 v1

**작성일**: 2026-04-23  
**기반**: 건설클레임 AI 보고서 검토 후 도출된 고도화 방향 논의  
**참조 문서**: `Docs/concept-claude-orchestrator.md`

---

## 현재 시스템 진단

### 강점
- n8n 자동 파이프라인으로 매일 지식 유입 (MiniFlux/ArXiv/yfinance)
- LightRAG 266개 문서 임베딩 — 시맨틱 검색 가능
- auto-linking으로 노트 간 자동 연결
- Notion 동기화, Linear 알림까지 완결된 자동화 루프
- Claude 온디맨드 접근: Zettelkasten MCP + /analyze 스킬

### 핵심 격차
| 격차 | 현재 | 목표 |
|------|------|------|
| 연구 방식 | 수동 수집 → 저장 | 질문 → 탐색 → 비판 → 종합 → 재탐색 |
| 검색 품질 | DuckDuckGo (낮음) | Tavily + Google CSE (높음) |
| 리서치 깊이 | 단일 엔진, 단일 보고서 | 다중 엔진, 교차 검증, 반복 심화 |
| 도메인 지식 | 일반 뉴스·논문 | 계약서·표준·판례 등 전문 문서 |
| 프로젝트 공간 | 없음 | `03-project/` 구조화 |
| Claude 결과 저장 | 세션 종료 후 소실 위험 | 반드시 Vault에 기록 |

## 설계 원칙

### 1. 이중 레이어 구조 (변경 없음)
```
[자동 레이어] n8n 07:00 — Claude 개입 없음
  수집 → 요약 → 임베딩 → 연결 → Notion → Linear

[온디맨드 레이어] 사용자 요청 시에만
  Claude → Vault 읽기/쓰기 → 리서치 → 분석 → Vault 저장
  (토큰 비용 발생 → 사용자가 범위 제어)
```

### 2. 프로젝트 지식 공간 — `03-project/` 폴더
물리적 격리보다 **Vault 내 구조화**가 더 강력하다.  
Vault 안에 있어야 LightRAG 그래프와 auto-linking이 연결된다.

```
ObsiVault/03-project/
  └── {프로젝트명}/              예: construction-claim-ai/
        _overview.md             프로젝트 정의, 핵심 질문, 진행 상태
        _questions.md            연구 질문 목록 + 답변 상태 추적
        research-{N}-{엔진}.md   리서치 보고서 (GPT Researcher / Perplexity)
        critic-{N}.md            비판 보고서 (Gemini Flash)
        synthesis-{N}.md         종합 노트 → 완성 시 02-permanent/ 승격
```

시간이 지나면 n8n 자동 파이프라인이 새 논문·뉴스를 LightRAG에 추가하고,  
`03-project/`의 노트들과 자동 연결된다. 프로젝트 지식이 수동 개입 없이 풍부해진다.

## 구현 로드맵

### Phase A — 즉각 효과 (단순 설정 변경)

#### A-1. 검색 엔진 교체: Tavily + Google CSE 이중화
**효과**: DuckDuckGo 대비 리서치 품질 대폭 향상. 설정 변경만으로 완료.

| 엔진 | 무료 한도 | 용도 |
|------|----------|------|
| Tavily | 1,000쿼리/월 | n8n 자동 파이프라인, 일상적 리서치 |
| Google Custom Search API | **100쿼리/일** | 중요 프로젝트 심층 리서치 |

Google CSE 설정:
1. [Google Cloud Console](https://console.cloud.google.com) → Custom Search API 활성화
2. [Programmable Search Engine](https://programmablesearchengine.google.com) → CSE 생성
3. `.env` 추가:
   ```
   TAVILY_API_KEY=tvly-...
   GOOGLE_API_KEY=AIza...
   GOOGLE_CX_ID=...
   ```
4. GPT Researcher 기본: `RETRIEVER=tavily`  
   중요 리서치 시: `RETRIEVER=google` (Claude가 온디맨드로 선택)

#### A-2. Perplexity API 연결 준비
**효과**: 사실 기반 신뢰도 가장 높은 엔진 추가. `.env`만 추가하면 준비 완료.

```
PERPLEXITY_API_KEY=pplx-...
```

> **참고**: Perplexity Pro($20/월)는 앱 전용 Deep Research다.  
> 자동화에 필요한 것은 **pplx-api** (별도 과금, `sonar-deep-research` 모델).  
> API 없이 앱에서 수동 실행 후 결과를 복사하는 방식으로도 활용 가능.

n8n 엔진 선택 분기:
- 기본: GPT Researcher (Tavily)
- 중요 질문: Perplexity API 또는 Google CSE
- Claude가 온디맨드로 엔진 선택 후 실행 지시

#### A-3. GPT Researcher → Vault 자동 저장 스크립트
**효과**: 리서치 결과가 LightRAG에 자동 임베딩됨. 기존 자산과 연결.

```python
# scripts/save-research.py
# gpt-researcher-outputs/ → 03-project/{프로젝트명}/research-{N}.md
# frontmatter 자동 추가 → LightRAG 임베딩 큐
```

실행 흐름:
```
GPT Researcher 보고서 생성 (localhost:8888)
  → save-research.py 실행 (Claude 온디맨드 지시)
  → 03-project/{프로젝트명}/research-{N}-{엔진}.md 저장
  → n8n 자동 파이프라인이 다음날 LightRAG 임베딩
  → auto-linking이 기존 노트와 연결
```

### Phase B — 리서치 품질 향상 (Researcher + Critic Bot)

**목표**: 단일 보고서의 한계를 반복적 비판과 보완으로 극복

#### B-1. 반자동 증폭 리서치 사이클

Python 스크립트(`scripts/research-cycle.py`)로 구현.  
n8n보다 Python이 적합한 이유: 조건부 반복 루프 제어가 용이.

```
사용자: 질문 + 사이클 수 입력 (기본 N=2)
  ↓
[Researcher Bot] GPT Researcher / Perplexity → 초안 보고서
  → 저장: research-{N}-draft.md
  ↓
[Critic Bot] Gemini Flash (무료 티어) → 비판 보고서
  — 빠진 관점, 약한 근거, 상충하는 주장, 검증 필요 출처
  → 저장: critic-{N}.md
  ↓
[Researcher Bot] 비판 기반 보완 리서치 → 개선 보고서
  → 저장: research-{N+1}-improved.md
  ↓
(N회 반복)
  ↓
사용자 최종 검토 → synthesis.md 승격
```

**비용 구조**:
- Researcher Bot: DeepSeek V3(저렴) 또는 Perplexity API
- Critic Bot: Gemini Flash 무료 티어 → **추가 비용 없음**
- Claude: 오케스트레이션 지시만 (토큰 최소화)

#### B-2. 다중 엔진 교차 검증 (선택적)

같은 질문을 두 엔진에 동시 실행 후 차이를 분석:
```
질문 → GPT Researcher (Tavily 또는 Google CSE)
     → Perplexity API (sonar-deep-research)
     ↓
Critic Bot: 두 보고서 비교 → 불일치 지점 명시
     ↓
synthesis.md: 교차 검증된 종합 노트
```

### Phase C — 도메인 특화 문서 수집 파이프라인

**배경**: 건설 계약서(FIDIC 등), 법령, 표준, 공공 지침 같은 문서들은  
양이 많고 당장 쓸모는 없지만 특정 프로젝트에서 핵심 자료가 된다.  
미리 파싱해서 LightRAG에 넣어두면, 질의 시 즉시 반환된다.

#### C-1. 3층 수집 구조

```
[수집층]  scripts/crawl-domain.py
  Playwright → 대상 사이트 크롤링
  → PDF 다운로드 → 07-archive/{source}/{version}/원본.pdf 보관
  → 원본은 절대 수정하지 않음 (증거 능력 보존)

[파싱층]  scripts/parse-domain-pdf.py
  PyMuPDF + pdfplumber → 텍스트·표 추출
  → 청킹 기준: 조항 번호 체계 (예: FIDIC 20.1, 건기법 시행령 제X조)
    (토큰 단위 청킹 금지 — 조항이 잘리면 의미 훼손)
  → 메타데이터 부착:
      doc_name, issuer, version, published_date, article_id, article_title

[임베딩층]  기존 embed-vault.py 활용
  → LightRAG에 청크 단위 임베딩
  → 질의 시 "FIDIC 20.1조" → 해당 조항 전문 즉시 반환
```

#### C-2. Lazy Loading 전략

전체 문서를 지금 임베딩할 필요 없다.

```
1단계 (즉시): 목차 + 조항 제목만 임베딩
  → 질의에서 관련 조항 식별

2단계 (온디맨드): 히트된 조항의 전문을 임베딩
  → Claude가 필요 시 실행 지시
  → 이후 재질의 시 전문 반환
```

**효과**: 임베딩 비용·시간 최소화, 필요한 부분만 정밀 로딩

#### C-3. 버전 관리

버전 혼용 시 오답 위험이 크다.  
예: FIDIC 1999 Red Book 20.1조 ≠ FIDIC 2017 Edition 20.2조

```
메타데이터: version: "FIDIC-1999-Red-Book"
LightRAG 질의 시 버전 필터 명시:
  "FIDIC 2017 기준 클레임 통지 기한은?"
  → version: FIDIC-2017 필터 적용
```

#### C-4. 우선 수집 대상 (건설 도메인 예시)

| 문서 | 출처 | 형식 | 우선순위 |
|------|------|------|----------|
| FIDIC 계약 조건 (1999/2017) | fidic.org | PDF | 높음 |
| NEC4 계약 조건 | neccontract.com | PDF | 높음 |
| 건설기술진흥법 시행령 | law.go.kr | HTML/PDF | 높음 |
| 표준시방서 | 국토부 | PDF | 중간 |
| 건설분쟁 조정 사례 | 건설분쟁조정위원회 | PDF | 중간 |
| KPMG/McKinsey 건설 리포트 | 각 사이트 | PDF | 낮음 |

### Phase D — Claude 온디맨드 통합 (장기)

현재 Zettelkasten MCP가 설치됐으나 Claude Code 재시작 후 활성화가 필요하다.  
활성화 확인 후 아래 시나리오들을 순차 구현한다.

#### 온디맨드 시나리오 1: 프로젝트 리서치 시작
```
사용자: "건설클레임 AI 에이전트 요구사항 리서치해줘 (3사이클)"
  ↓
Claude:
  1. _questions.md 작성 (세부 질문 5-7개 분해)
  2. research-cycle.py 실행 (N=3)
  3. 결과를 03-project/construction-claim-ai/ 저장
  4. /analyze로 기존 LightRAG 지식과 교차 확인
  5. synthesis.md 초안 작성
```

#### 온디맨드 시나리오 2: 도메인 문서 수집
```
사용자: "FIDIC 2017 수집하고 싶어"
  ↓
Claude:
  1. crawl-domain.py 실행 (대상 URL 지정)
  2. parse-domain-pdf.py 실행
  3. 목차 먼저 임베딩 (Lazy Loading 1단계)
  4. 임베딩 완료 보고
```

#### 온디맨드 시나리오 3: 기존 지식과 새 리서치 연결
```
사용자: "/analyze 건설클레임에서 AI 인과관계 분석 가능성"
  ↓
Claude:
  LightRAG에서 관련 노트 검색
  → 03-project/ + 01-literature/ + 04-gemini/ 교차
  → 종합 분석 보고서 생성
  → 00-fleeting/ 또는 03-project/에 저장 (세션 후에도 보존)
```

## 미결 설계 질문 (향후 결정 필요)

1. **Perplexity API vs 앱 수동 사용**: API 비용 대비 가치 판단 필요  
   → 당장은 앱 수동 실행 후 결과 복사로 운용 가능

2. **04-gemini 노트의 Notion 동기화**: 244개 파일을 Notion에 올릴지 여부  
   → LightRAG에는 이미 임베딩됨. Notion은 선택적으로 결정

3. **리서치 사이클 사이클 수 기본값**: N=2가 비용과 품질의 적정 균형인지  
   → 첫 실행 후 사용자 피드백으로 조정

4. **도메인 크롤링 시작 프로젝트**: 건설클레임부터 시작할지 다른 도메인부터 할지
