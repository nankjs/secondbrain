## 1. 개요

이 문서는 세컨드브레인 프로젝트의 **범위 경계**와 **제약 조건**을 명확히 한다.

- **포함 범위**: MVP에 포함되는 기능
- **제외 범위**: 명시적으로 제외하는 기능
- **기술 제약**: 아키텍처/기술 선택의 한계
- **운영 제약**: 비용, 인력, 시간
- **비즈니스 제약**: 모델 및 전략 관련

---

## 2. 포함 범위 (In Scope)

### 기능 범위 (MVP)

#### Phase A: Obsidian Vault 구조화 ✓
- Obsidian Vault 폴더 구조 설계
- Frontmatter 스키마 정의
- Claude Code 4개 스킬 (/fleeting, /literature, /lint, /index)
- Git 저장소 관리
- 템플릿 작성

#### Phase B: n8n 자동 수집 ✓
- 3개 도메인 (기독교, 학술, 경제) 자동 수집
- 신뢰 소스 20개 선정 및 통합 (RSS, API)
- 주 1회 자동 실행 워크플로우
- Claude API로 원문 → 마크다운 변환
- Obsidian 자동 저장
- Slack 알림

#### Phase C: 지식 연결 ✓
- LightRAG 벡터 저장소 구축
- 자동 유사도 검색 (Top 20)
- Claude MCP 관계 분석
- Obsidian [[링크]] 자동 생성
- 사용자 피드백 메커니즘 (선택)

#### Phase D: 즉시 분석 ✓
- 한 줄 질문 수신
- 벡터 검색 → 관련 노트 추출 (Top 30)
- Claude API → 분석 보고서 생성
- 마크다운 포맷 출력
- 5분 이내 응답 (목표)

#### 지원 기능
- Git 백업 (주 1회)
- 에러 로깅 및 모니터링
- Docker Compose 자동화
- 환경변수 관리 (.env)

### 사용자 범위 (Initial)
- **개발자** (1명, 자신)
- **목사님** (협력, 수동 온보딩)
- **향후 확장 가능** (대학생, 디자이너, 사무원 — Phase E+)

### 데이터 범위
- **저장소**: Obsidian Vault (로컬 마크다운)
- **메타데이터**: SQLite (노트 인덱스)
- **벡터 DB**: LightRAG (임베딩)
- **백업**: Git + 월간 수동 백업

### 배포 환경
- **개발/운영**: Windows 11 PC (로컬)
- **컨테이너**: Docker Desktop (n8n, LightRAG, MiniFlux)
- **향후 클라우드**: 6개월 후 검토 (현재 제외)

## 3. 제외 범위 (Out of Scope) — MVP

### 명시적으로 제외하는 기능

#### 모바일 앱
**이유**: Windows PC 로컬 운영이 기본 → 모바일은 향후 Web UI로 지원
**대안**: Web UI (FastAPI + React) — Phase E+ 계획

#### 실시간 협업
**이유**: MVP는 개인 + 반자동 팀 모델 → 실시간 동기화 복잡도 높음
**대안**: Notion 공유 (비동기) — Phase E+ 계획

#### 소셜 미디어 자동 발행
**이유**: Threads, X, LinkedIn 자동 포스팅 복잡도 높음 + 개인정보 위험
**대안**: 수동 공유 (메뉴얼로만)

#### 동시 다사용자 실시간 편집
**이유**: Obsidian 기본 구조가 로컬 단일 파일 → 실시간 동기화 불가
**대안**: Notion 읽기 전용 공유 + 주기적 싱크

#### 외부 공개 (Quartz)
**이유**: 낮은 우선순위 + Phase F 이후
**일정**: 6개월 후 검토 (현재 제외)

#### 유료 구독 모델
**이유**: MVP는 개인 + 팀 자산 (수익화 아님)
**향후**: 1년 후 팀 규모 확대 시 검토

#### 권한 기반 접근 제어 (RBAC)
**이유**: Phase E까지 단순 역할별 공유로 충분
**향후**: 다중 팀 지원 시 (2년 후)

#### 다국어 지원
**이유**: MVP는 한국어 + 영어 (논문)만 지원
**향후**: 필요시 추가

## 4. 기술 제약

### 아키텍처 제약

#### Obsidian Vault 단일 저장소
| 제약 | 영향 | 대응 |
|------|------|------|
| 실시간 동기화 불가 (로컬 파일) | 여러 PC에서 동시 편집 불가 | Git으로 비동기 싱크 |
| 용량 한계 (1GB 초과시 성능 저하) | 장기 운영 시 아카이빙 필요 | 분기별 아카이빙 (07-archive/) |
| 경합 (Conflicts) 가능 | 동기화 중 충돌 | 수동 병합 + 프로세스 개선 |

#### LightRAG 벡터 DB
| 제약 | 영향 | 대응 |
|------|------|------|
| 임베딩 차원 고정 (1536) | 다른 모델 사용 불가 | OpenAI 모델만 지원 |
| 벡터 검색 정확도 (0~1) | 완벽한 관련성 보장 불가 | 사용자 피드백으로 학습 |
| 메모리 사용 증가 | 노트 1000개↑ 성능 저하 가능 | 인덱싱 최적화 + 캐싱 |

#### Claude API 의존성
| 제약 | 영향 | 대응 |
|------|------|------|
| API 비용 발생 | 월 $30~50 예상 | 사용량 모니터링 |
| Rate Limiting | 대량 요청시 대기 | 배치 처리, 큐 관리 |
| 모델 업데이트 | 예상치 못한 동작 변화 | 버전 고정, 정기 테스트 |

### 성능 제약

| 지표 | 목표 | 현실적 한계 | 개선 계획 |
|------|------|----------|---------|
| 벡터 검색 시간 | < 1초 | 500개 노트 기준 | 인덱싱 최적화 |
| 보고서 생성 | < 5분 | 토큰 수 많으면 10분 | 컨텍스트 압축 |
| 자동 연결 처리 | < 5분/노트 | 복잡도 높으면 10분 | 배치 처리 |

## 5. 운영 제약

### 리소스 제약

#### 인력
| 역할 | 투입 | 제약 |
|------|------|------|
| 개발자 (주담당) | 1명 full-time | MVP 9주, 이후 유지보수만 |
| 설계 리뷰 | 자신 | 자체 검증 → 객관성 부족 위험 |

**대응**:
- Phase 별 중간 검토 (주 1회)
- 사용자 피드백 정기 수집 (월 1회)

#### 시간
| Phase | 예상 기간 | 여유 | 위험도 |
|-------|---------|------|--------|
| A | 2주 | +1주 | 낮음 |
| B | 2주 | +1주 | 높음 (우선) |
| C | 2주 | +1주 | 중간 |
| D | 2주 | +1주 | 중간 |
| E | 1주 | 없음 | 낮음 |
| F | 1주 | 없음 | 낮음 |

**대응**:
- Phase 우선순위: B > C > D > E > F
- 시간 초과시: E, F 연기 (MVP 품질 우선)

#### 비용
| 항목 | 월간 | 3개월 | 12개월 |

| 인프라 (Docker, Git) | 무료 | 무료 | 무료 |
| 클라우드 (향후) | - | - | $100+ |
| **합계** | **$35** | **$105** | **$420+** |

**제약**: 개인 비용 부담 → 월 $100 이내 유지

## 8. 성능 및 확장성 제약

### 초기 성능 기준
```
노트 수:    100 ~  500개 (3개월)
응답 시간:  < 5분 (목표)
벡터 검색:  < 1초
저장소:     < 500MB

한계:
- 1000개↑ 노트: 성능 저하 가능
- 동시 API 요청 10개↑: 대기 가능
```

### 확장 로드맵
| 시점 | 제약 | 대응 |
|------|------|------|
| 3개월 | 100개 노트 | 현재 아키텍처 OK |
| 6개월 | 500개 노트 | 인덱싱 최적화 필요 |
| 12개월 | 1000개↑ 노트 | 클라우드 이전 검토 |

## 9. 제외하는 사용 시나리오

### 시나리오 A: 실시간 다중 사용자 협업
```
❌ 제외: 같은 노트를 여러 명이 동시에 편집
이유: Obsidian 로컬 구조 한계

✓ 대안: Notion 읽기 전용 공유 (Phase E+)
```

### 시나리오 B: 모바일 앱에서 노트 편집
```
❌ 제외: iPhone/안드로이드 앱 (MVP)
이유: 모바일 앱 개발 복잡도 높음

✓ 대안: Web UI (FastAPI + React) — Phase E+
```

### 시나리오 C: 자동 소셜 발행
```
❌ 제외: 분석 결과를 자동으로 Threads/X에 포스팅
이유: 개인정보 + 맥락 손실 위험

✓ 수동 방식: 결과 복사 → 직접 포스팅
```

### 시나리오 D: AI가 사용자 피드백 완전 학습
```
❌ 제외: "틀렸어" 피드백 → AI가 자동 재학습
이유: 프롬프트 기반 설계 (가중치 재학습 필요)

✓ 대안: 정기 프롬프트 개선 (월 1회)
```

### 시나리오 E: 다양한 언어 지원
```
❌ 제외: 중국어, 일본어, 스페인어 등
이유: 임베딩 모델 선택에 영향 + 메타데이터 복잡화

✓ 지원: 한국어 + 영어 (논문)
```

## 12. 변경 관리

### 범위 추가 요청 시 프로세스

```
요청 접수
    ↓
영향도 평가
    ├─ 기술 영향도: 낮음/중간/높음
    ├─ 일정 영향도: 일 수
    └─ 비용 영향도: $
    ↓
우선순위 검토
    ├─ 필수 (MVP 완성도): 즉시 반영
    ├─ 중요 (3개월): Phase 계획
    └─ 선택 (6개월+): 대기
    ↓
의사결정
    ├─ 승인 → Phase 계획에 추가
    └─ 거절 → 후속 검토 일정 설정
```

### 범위 변경 승인 권한
- **MVP 완성까지**: 자신이 단독 결정
- **3개월 후**: 목사님, 대학생과 협의
- **6개월 후**: 공식 팀 회의 (분기별)
